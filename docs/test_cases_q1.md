## Top 3 Test Cases — Priority Justification (Q1 Part 2)

**TC-01 — Successful end-to-end booking with valid Stripe payment**  
This is the core revenue-generating transaction for Ezra's business. If a valid card is rejected or the confirmation page fails to display correct booking details, the company loses a paying customer immediately and trust in the platform collapses. Every other test in this suite depends on this happy path working correctly — it is the absolute baseline of the entire booking system. A failure here is a P0 incident.

**TC-02 — Payment declined — invalid card rejected gracefully**  
Payment failures are the second most common real-world scenario after a successful payment. A silent failure, unhandled exception, or unclear error message here can cause double-charges, orphaned booking records in the database, or member confusion about whether they are booked. For a medical service handling sensitive health appointments, this is unacceptable. The error path must be as robust and well-tested as the happy path.

**TC-03 — Enforcing exactly 3 time slots before proceeding from Step 2**  
The 3-slot requirement is a hard operational constraint tied directly to how Ezra's imaging center partners confirm appointments. If this validation is bypassed — either through a UI bug or a direct API call — it breaks the downstream scheduling workflow for staff, can result in unconfirmed appointments, and leads to member no-shows or wasted imaging slots. This test ensures the business rule is enforced at the UI level and cannot be circumvented.
